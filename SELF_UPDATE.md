# Maurice Self-Update

## Principle

Maurice may help improve itself, but normal agents must not directly modify the runtime root.

Runtime changes are high-trust actions.
They go through a proposal workflow.


## Protected Targets

Protected runtime targets include:

- kernel code
- host code
- system skills
- packaged defaults
- migration code
- installer and service files

Normal workspace-scoped tools must not write to these targets.


## Proposal Workflow

When an agent identifies a runtime or system skill improvement, it should create a proposal under the workspace.

Suggested shape:

```text
$workspace/proposals/runtime/<proposal_id>/
  proposal.yaml
  summary.md
  patch.diff
  risk.md
  test_plan.md
```

The proposal must be user-readable and host-applicable.


## Proposal Record

`proposal.yaml` should use this shape:

```yaml
id: "runtime_2026_04_26_memory_fix"
created_at: "2026-04-26T00:00:00Z"
created_by_agent: "main"
target:
  type: "kernel|host|system_skill|packaged_defaults|installer"
  name: "system_skill:memory"
  runtime_path: "$runtime/system_skills/memory"
permission:
  class: "runtime.write"
  mode: "proposal_only"
summary: "Fix memory migration edge case."
risk: "low|medium|high"
requires_restart: true
patch: "patch.diff"
test_plan: "test_plan.md"
status: "draft|pending_review|approved|rejected|applied|failed"
```


## Agent Responsibilities

An agent may:

- inspect selected runtime docs or manifests when policy allows
- create a proposal in the workspace
- explain the reason and risk
- provide a patch
- provide a test plan
- update the proposal before approval

An agent must not:

- write directly into runtime root
- replace a system skill in place
- apply its own proposal
- hide risk or omit touched files
- treat `power` profile as direct runtime write permission


## User And Host Responsibilities

The user approves or rejects the proposal.

The host applies approved proposals.

The host should:

- verify the proposal target
- verify the patch only touches approved runtime paths
- run the declared test plan when possible
- backup touched files or require version control
- apply the patch
- emit audit events
- restart affected services when required


## Approval Rules

Runtime proposals use `runtime.write`.

Default mode is:

```yaml
class: "runtime.write"
decision: "ask"
scope:
  mode: "proposal_only"
rememberable: false
```

Direct apply mode is not a normal agent capability.

If direct apply is ever supported, it must require:

- explicit user command
- exact target scope
- exact patch hash
- audit event
- rollback plan


## Events

Self-update should emit structured events:

- `runtime.proposal_created`
- `runtime.proposal_updated`
- `runtime.proposal_approved`
- `runtime.proposal_rejected`
- `runtime.proposal_applied`
- `runtime.proposal_failed`

Events should include:

- proposal id
- creating agent
- target type
- target name
- patch hash
- status


## Failure Behavior

If application fails, the host should:

- stop applying further changes
- record the failed state
- preserve logs
- restore from backup when possible
- mark the proposal `failed`
- emit `runtime.proposal_failed`

The agent may then propose a follow-up fix, but it still cannot write directly to runtime.


## Development Mode

A development installation may allow more direct runtime edits.

Development mode must be explicit.
It should not be enabled by selecting the `power` profile alone.

Even in development mode, proposals are preferred for changes generated by agents.
